Databricks Python CI/CD Architecture Guide

Best Practices with Databricks Asset Bundles (DAB), Jobs, and Multi-Env Deployments

1. Pipeline vs. Jobs – Clearing the Confusion

While both components run code artifacts, their design architectural patterns serve entirely different purposes:

Feature Databricks Jobs (Workflows) Delta Live Tables (DLT) Pipelines
Primary Role A general-purpose task orchestrator (DAG). Manages when and what runs. A declarative framework purpose-built for data streaming, ingestion, and robust ETL.
Supported Formats Notebooks, Python Scripts, Wheel files, DLT Pipelines, SQL, dbt, etc. Exclusively Python/SQL structures utilizing the dlt framework library.
Infrastructure Triggers workloads across customizable clusters (classic, job, or serverless). Abstracts compute, dynamically managing scaling and underlying schema changes automatically.

2. Workspaces & Git Repository Checkouts

The enterprise best practice is to separate your lifecycle environments into two distinct Databricks Workspaces (e.g., a Dev Workspace and a Prod Workspace) to guarantee strict security and data isolation boundaries.

Where should code be checked out?

  • Dev Workspace: Developers clone the corporate Git repository directly into their personal sandbox area under /Workspace/Users/user@company.com/projects/... inside Git Folders (formerly Repos) for active engineering, testing, and debugging.
  • Prod Workspace: Never perform manual Git operations or checkouts on production infrastructure. Production workflows rely entirely on automated continuous integration runners. The CI/CD system packages, bundles, and pushes frozen code artifacts into internal storage or Unity Catalog Volumes, referencing files securely by distinct deployment versions.

3. Understanding Databricks Asset Bundles (DAB)

Databricks Asset Bundles (DAB) represent the official Infrastructure-as-Code (IaC) implementation tool tailored specifically for Databricks ecosystem deployments. Similar to how Terraform structures underlying cloud services, a single databricks.yml template explicitly defines your target environments, resource schemas, variables, and step sequences.

When running a deployment, DAB automatically compresses local script packages, transports them securely to targeted environment directories, and hooks up the cloud jobs seamlessly via native APIs.

4. Multi-Engineer Parallel Git Workflow

When 3 engineers (Anna, Ben, and Cecil) contribute to the identical code repository concurrently within the Dev Workspace, DAB development mechanics ensure isolation:

  1. Separate Checkouts: Every contributor pulls down code into their localized directory spaces:
    • /Workspace/Users/anna@company.com/hello-world-pipeline
    • /Workspace/Users/ben@company.com/hello-world-pipeline
    • /Workspace/Users/cecil@company.com/hello-world-pipeline
  2. Execution Sandbox: When Anna makes modifications to a local step and executes databricks bundle deploy -t dev, DAB processes her unique auth token. Because the target is configured with mode: development, DAB flags the deployment as isolated, appending her identity onto the resource names (e.g., creating dev_hello_world_workflow_anna).
  3. Collision Prevention: Ben and Cecil run identical terminal operations concurrently. Their adjustments compile cleanly into separate testing workflows without intercepting, modifying, or breaking Anna's isolated pipeline runs.

5. Concrete Multi-Task databricks.yml Definition

Below is the specific structural configuration specifying how a 3-task continuous chain is modeled using clean relative paths:

bundle:
  name: hello-world-pipeline

targets:
  dev:
    workspace:
      host: https://adb-dev.azuredatabricks.net
    mode: development 

  prod:
    workspace:
      host: https://adb-prod.azuredatabricks.net
    mode: production

resources:
  jobs:
    hello_world_job:
      name: "HelloWorld_Workflow_${bundle.target}"
      
      tasks:
        # Task 1: Ingestion Entry Point
        - task_key: run_hello
          new_cluster:
            spark_version: "14.3.x-scala2.12"
            node_type_id: "Standard_D4s_v5"
            num_workers: 1
          spark_python_task:
            python_file: ./src/task1_hello.py

        # Task 2: Processing (Dependent on Task 1)
        - task_key: run_world
          depends_on:
            - task_key: run_hello
          new_cluster:
            spark_version: "14.3.x-scala2.12"
            node_type_id: "Standard_D4s_v5"
            num_workers: 1
          spark_python_task:
            python_file: ./src/task2_world.py

        # Task 3: Reporting (Dependent on Task 2)
        - task_key: run_summary
          depends_on:
            - task_key: run_world
          new_cluster:
            spark_version: "14.3.x-scala2.12"
            node_type_id: "Standard_D4s_v5"
            num_workers: 1
          spark_python_task:
            python_file: ./src/task3_summary.py
How relative files are mapped: Notice the python_file: ./src/task1_hello.py property. When building your target states, DAB automatically picks up the local directory paths, streams them safely into target storage structures (e.g., workspace roots or private enterprise Volumes), and replaces them dynamically with absolute internal paths (e.g., /Workspace/Shared/.bundle/...) to ensure the job runs flawlessly without any real-time Git access dependencies.

6. Environmental Parameter Management

To safely isolate environment states without creating complex hardcoded code branches, variables are structured into global blueprints, overrode at the execution target block, and passed down into tasks via standard runtime environment arrays.

Advanced databricks.yml Parameter Integration

bundle:
  name: hello-world-pipeline

# Define the variable blueprint variables globally
variables:
  storage_account_name:
    type: string
  subscription_id:
    type: string
  sp_client_id:
    type: string
  secret_scope_name:
    type: string

targets:
  dev:
    workspace:
      host: https://adb-dev.azuredatabricks.net
    mode: development
    variables:
      storage_account_name: "mystorageaccountdev"
      subscription_id: "11111111-2222-3333-4444-555555555555"
      sp_client_id: "aaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee"
      secret_scope_name: "dev-secrets-scope"

  prod:
    workspace:
      host: https://adb-prod.azuredatabricks.net
    mode: production
    variables:
      storage_account_name: "mystorageaccountprod"
      subscription_id: "99999999-8888-7777-6666-555555555555"
      sp_client_id: "zzzzzzz-yyyy-xxxx-wwww-vvvvvvvvvvvv"
      secret_scope_name: "prod-secrets-scope"

resources:
  jobs:
    hello_world_job:
      name: "HelloWorld_Workflow_${bundle.target}"
      tasks:
        - task_key: run_summary
          new_cluster:
            spark_version: "14.3.x-scala2.12"
            node_type_id: "Standard_D4s_v5"
            num_workers: 1
            
            # Injecting variables as OS environment handles to the cluster
            spark_env_vars:
              STORAGE_ACCOUNT: ${var.storage_account_name}
              SUB_ID: ${var.subscription_id}
              SP_CLIENT_ID: ${var.sp_client_id}
              SECRET_SCOPE: ${var.secret_scope_name}
              
          spark_python_task:
            python_file: ./src/task3_summary.py

7. Consuming Variables and Secrets inside Python

Sensitive data points (such as the Service Principal's secret password) are never hardcoded or stored inside text configurations. Instead, they are referenced securely through the Databricks Secret Scope Vault, pulled down dynamically on-the-fly during active cluster compute lifecycles:

import os
from pyspark.sql import SparkSession

# 1. Access non-sensitive configurations from environment handles
storage_account = os.environ.get("STORAGE_ACCOUNT")
subscription_id = os.environ.get("SUB_ID")
client_id = os.environ.get("SP_CLIENT_ID")
secret_scope = os.environ.get("SECRET_SCOPE")

spark = SparkSession.builder.getOrCreate()

# 2. Extract highly protected application passwords from the Secret Manager
client_secret = dbutils.secrets.get(scope=secret_scope, key="sp-client-secret")

# 3. Apply drivers configurations dynamically depending on target contexts
spark.conf.set(
    f"fs.azure.account.auth.type.{storage_account}.dfs.core.windows.net", 
    "OAuth"
)
spark.conf.set(
    f"fs.azure.account.oauth.provider.type.{storage_account}.dfs.core.windows.net", 
    "org.apache.hadoop.fs.azurebfs.oauth2.AzureADAuthenticator"
)
spark.conf.set(
    f"fs.azure.account.oauth2.client.id.{storage_account}.dfs.core.windows.net", 
    client_id
)
spark.conf.set(
    f"fs.azure.account.oauth2.client.secret.{storage_account}.dfs.core.windows.net", 
    client_secret
)
spark.conf.set(
    f"fs.azure.account.oauth2.client.endpoint.{storage_account}.dfs.core.windows.net", 
    f"https://login.microsoftonline.com/{subscription_id}/oauth2/token"
)

# Read or Write datasets smoothly across isolated targets
data_path = f"abfss://container-name@{storage_account}.dfs.core.windows.net/refined_analytics"
print(f"Executing decoupled cloud storage task targeting: {data_path}")

8. Automated CI/CD Execution Pipeline (GitHub Actions)

When engineering milestones are merged straight into the main code branch, the automated release pipelines pick up changes, run unit validations, authenticate as a secure Service Principal account identity, and apply the infrastructure mutations cleanly:

name: Production Deployment Automation

on:
  push:
    branches: [ main ]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout Codebase
        uses: actions/checkout@v4

      - name: Setup Databricks Asset CLI
        uses: databricks/setup-cli@v1

      - name: Execute Bundle Production Deploy
        env:
          DATABRICKS_HOST: https://adb-prod.azuredatabricks.net
          DATABRICKS_CLIENT_ID: ${{ secrets.PROD_SP_CLIENT_ID }}
          DATABRICKS_CLIENT_SECRET: ${{ secrets.PROD_SP_CLIENT_SECRET }}
        run: |
          databricks bundle deploy -t prod
